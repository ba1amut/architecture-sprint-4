# Выбор систем логирования
Список необходимых логов с уровнем **INFO** и систем они будут собираться
### Shop API
**Создание заказа**: время, идентификатор покупателя, номер заказа, статус.

**Обработка запроса**: время, метод HTTP, эндпоинт, статус ответа.
### CRM и CRM API
**Передача заказа в RabbitMQ**: время, номер заказа, статус передачи.

**Изменение статуса заказа в CRM**: время, номер заказа, новый статус.
### RabbitMQ
**Получение сообщения**: время, идентификатор сообщения, статус обработки.
### MES и MES API
**Расчёт стоимости заказа**: время, номер заказа, стоимость, статус.

**Начало производства**: время, номер заказа, статус.
### Базы данных (Shop DB, MES DB)
**Запись данных**: время, тип операции (CRUD), таблица, статус.
### 3D Files Storage
**Загрузка файла**: время, идентификатор файла, статус.

#### Использование других уровней логирования
##### Уровень DEBUG 
 Для отладки и поиска проблем в коде.
##### Уровень WARN
 Для предупреждений о потенциальных проблемах, которые не критичны, но требуют внимания.
##### Уровень ERROR
 Для ошибок, которые влияют на работу системы, но не приводят к её остановке.
##### Уровень FATAL
 Для критических ошибок, которые приводят к остановке системы.

# Мотивация
Вот основные причины, почему логирование необходимо:
1. **Диагностика проблем**:
    - Логи помогают быстро находить и устранять ошибки, что снижает время простоя системы.
2. **Улучшение производительности**:
    - Анализ логов позволяет выявить узкие места в системе (например, медленные запросы к базе данных) и оптимизировать их.
3. **Повышение надёжности системы**:
    - Логирование помогает предотвратить повторение ошибок, так как их причины можно проанализировать и устранить.
4. **Поддержка масштабирования**:
    - Логи помогают отслеживать нагрузку на систему и планировать её масштабирование.
5. **Улучшение пользовательского опыта**:
    - Быстрое устранение ошибок и оптимизация работы системы повышают удовлетворённость клиентов.

### Какие системы нужно настраивать в первую очередь

Поскольку команда не может реализовать логирование и трейсинг для всех систем одновременно, рекомендуется начать с ключевых компонентов, которые имеют наибольшее влияние на работу системы. Вот приоритетные системы:
1. Shop API
2. CRM API
3. MES API
   
Начиная с этих систем, компания сможет быстро получить значительные улучшения в работе системы и постепенно расширять логирование и трейсинг на другие компоненты.
# Предлагаемое решение
Для реализации решения по организации логирования будет использоваться ELK Stack (Elasticsearch, Logstash, Kibana).
- Elasticsearch: Поисковая система для хранения и индексации логов.
- Logstash: Инструмент для сбора, обработки и передачи логов в Elticsearch.
- Kibana: Визуализация и анализ логов.
#### Контроль доступа
- Аутентификация и авторизация: Использовать LDAP или OAuth2 для контроля доступа к Kibana и Elasticsearch.
- Разграничение прав: Только администраторы имеют доступ ко всем логам, разработчики — только к логам своих компонентов.
#### Шифрование данных
- Шифрование при передаче: Использовать HTTPS для защиты данных, передаваемых между компонентами.
- Шифрование при хранении: Шифрование данных в Elasticsearch с помощью плагинов
### Политика хранения логов
Организация отдельного индекс под систему хранения логов, определения и реализация срока хранения логов (удаление старых логов) ограничение размера логов (дополнительно возможно использование шардирования).

Эти меры обеспечат безопасное и эффективное логирование, что поможет компании улучшить производительность системы и оперативно реагировать на проблемы.

# Система анализа логов
Алертинг позволяет быстро выявлять и устранять проблемы, такие как ошибки, перегрузка системы или атаки.
#### Какие  алерты следует настроить
1. Высокий уровень ошибок.
2. Резкий рост нагрузки.
3. Долгое время выполнения запросов.
   
Настройку алертинга можно провести посредство ElastAlert, выявление аномалий через ML в Elasticsearch, визуализация в Kibana

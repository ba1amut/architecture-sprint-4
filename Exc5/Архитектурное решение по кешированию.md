С учётом описанных проблем (долгая загрузка дашборда MES, задержки в обработке заказов, перегрузка системы после открытия API), наиболее критичными узлами системы, которые стоит оптимизировать в первую очередь, являются:
### 1. MES (Manufacturing Execution System)
Кешировать данные о заказах и результаты расчёта стоимости 3D-моделей.
### 2. Shop API
Кешировать данные о заказах и запросов к базе данных.
### 3. CRM API
Кешировать данные о клиентах и списки заказов.

# Мотивация
Кэширование на MES, Shop API и CRM API поможет решить ключевые проблемы системы. В MES кэширование ускорит загрузку дашборда и расчёт стоимости заказов, особенно для сложных 3D-моделей. В Shop API кэширование снизит задержки в обработке запросов от клиентов и уменьшит нагрузку на базу данных. В CRM API кэширование ускорит работу менеджеров и передачу заказов в RabbitMQ. Эти меры улучшат производительность системы и повысят удовлетворённость клиентов и сотрудников.

# Предлагаемое решение
Для реализации предлагается  внедрить серверное кэширование, так как оно лучше подходит для решения описанных проблем. Серверное кэширование позволяет централизованно управлять кэшем. Оно также обеспечивает более гибкое управление данными и снижает нагрузку на базы данных и другие ресурсы.

Для серверного кэширования я предлагаю использовать паттерн **Cache-Aside**. Вот почему:

1. **Cache-Aside**:
    - **Почему подходит**:
        - Простота реализации: не требует сложной интеграции с базой данных.
        - Гибкость: можно кэшировать только те данные, которые действительно нужны.
        - Эффективность: снижает нагрузку на базу данных, так как запросы к ней происходят только при отсутствии данных в кэше.
            
    - **Почему другие паттерны не подходят**:
        - **Write-Through**: Требует синхронной записи в кэш и базу данных, что может замедлить работу системы, особенно при высокой нагрузке.
        - **Refresh-Ahead**: Сложнее в реализации и требует прогнозирования, какие данные понадобятся, что может быть неэффективно в текущей системе.
        - 
2. **Клиентское кэширование**:
    - **Почему не подходит**:
        - Клиентское кэширование (например, в браузере) не решает проблем с нагрузкой на серверы и базы данных.
        - Оно эффективно только для статических ресурсов (например, изображений, CSS), но не для динамических данных, таких как статусы заказов или результаты расчётов.
### Инвалидация 


 **Основная стратегия**: Инвалидация по ключу.
При изменении данных (например, статуса заказа) кэш для конкретного ключа очищается.

**Дополнительная стратегия:** Программная инвалидация.
После обновления данных в базе данных кэш очищается программно.
        
Преимущества:
    Точечная инвалидация, минимизация устаревших данных.
    Гибкость и точность управления кэшем.

**Временная инвалидация**:
    Не подходит, так как данные в системе часто обновляются, и важно избегать устаревших данных. Автоматическая очистка через TTL может привести к тому, что данные в кэше будут устаревшими до истечения срока.
    
**Инвалидация по событию**:
    Подходит, но требует дополнительной инфраструктуры для обработки событий (например, message broker). Это усложняет систему и увеличивает затраты на поддержку.
